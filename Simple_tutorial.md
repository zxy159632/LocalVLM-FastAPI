# 从零到一：用 vLLM + FastAPI + 网页前端搭一个可在手机访问的本地大模型服务

这是一份面向初学者的完整教程。

目标不是只让你“把命令敲通”，而是让你真正明白：

- 模型服务到底是什么
- 为什么要在 vLLM 外面再包一层 FastAPI
- 为什么前端要拆成文本 / 文件 / 图片三个入口
- 为什么手机浏览器能访问你电脑上的本地模型
- 这套项目结构为什么适合写进简历

如果你是准备从事**推理加速 / 大模型部署工程师**方向，这个项目很适合作为一个入门级但很完整的工程样例。

---

# 二、先理解整个系统

很多初学者一上来就开始复制命令，结果虽然能跑，但完全不知道自己搭了什么。你可以把这个项目理解成 3 层。

## 第 1 层：vLLM

这是模型真正运行的地方。

它负责：

- 加载模型权重
- 占用 GPU 显存
- 接收 OpenAI 风格请求
- 返回模型生成结果

你可以把它理解成：**模型发动机**。

## 第 2 层：FastAPI

这是业务接口层。

它负责：

- 把一个通用的模型能力，整理成几个清晰接口
- 做参数接收
- 统一响应格式
- 处理中间逻辑

你可以把它理解成：**模型发动机外面的一层控制台**。

## 第 3 层：前端网页

这是用户操作层。

它负责：

- 输入文本
- 读取文件
- 压缩图片
- 调后端接口
- 展示模型返回结果

你可以把它理解成：**用户看得见、摸得着的界面**。

---

# 三、为什么不能只用 vLLM

你可能会问：

“既然 vLLM 已经能提供 OpenAI 兼容接口了，为什么还要自己写 FastAPI？”

这是因为两者职责不一样。

## 只用 vLLM 时

你得到的是一个很底层的能力：

- 模型能回答问题
- 接口格式是通用的 `/v1/chat/completions`

但它并不关心：

- 你的业务到底分成文本、文件还是图片
- 你希望返回什么字段
- 你想怎么做健康检查
- 你想怎么组织前端页面

## 加一层 FastAPI 后

你就能把它整理成真正像项目的样子：

- `/api/chat/text`
- `/api/chat/file`
- `/api/chat/image`
- `/health`
- `/`

这样做后，别人一看你的项目就明白：

- 文本能力在哪里
- 文件能力在哪里
- 图片能力在哪里

这对演示、简历、面试、后续安卓接入都更友好。

# 五、准备模型

这一步的核心不是“下载任意一个模型”，而是选择一个**你的显卡能带得动**的模型。

当前项目使用的是一个量化后的 Qwen 模型：

```text
.../models/Qwen3.5-0.8B-int4-AutoRound
```

这类量化小模型非常适合作为本地演示起点，因为：

- 显存压力更小
- 启动更稳
- 响应更快
- 更适合笔记本电脑

项目初期其实经历过完整的一轮试错：先跑 vLLM 基线，再尝试 7B、AWQ、显存优化、框架替换，最后逐渐沉淀出“消费级显卡部署时要控制显存利用率、上下文长度和内存策略”的经验。

这对初学者非常重要：

> 不是任何模型都适合一上来就部署，第一目标应该是“先找一个能稳定跑通的模型”，再谈更复杂的性能优化。

# 六、第一个阶段：先单独启动 vLLM

先不急着写 FastAPI。先做最底层验证：**模型能不能独立启动**。

## 启动命令示例

```bash
vllm serve {path_to_model}/Qwen3.5-0.8B-int4-AutoRound \
  --host 0.0.0.0 \
  --port 8001 \
  --served-model-name qwen3.5-int4 \
  --quantization autoround \
  --gpu-memory-utilization 0.75 \
  --max-model-len 4096 \
  --tensor-parallel-size 1 \
  --disable-frontend-multiprocessing \
  --enforce-eager \
  --trust-remote-code \
  --allowed-origins '["*"]'
```

## 参数理解

### `--served-model-name`

给模型起一个对外服务名。这样你请求时写：

```json
"model": "qwen3.5-int4"
```

而不是写一长串本地绝对路径。

### `--gpu-memory-utilization 0.75`

让 vLLM 不要一上来吃满显存。

在资源受限的本地环境里，这个参数很实用。

### `--max-model-len 4096`

不要让系统默认按极长上下文去预留 KV Cache。

上下文越大，KV Cache 预留越多，显存压力越大。

### `--enforce-eager`

禁用 CUDA Graph。

这么做的原因很现实：**更省显存，启动更稳**。

这类“消费级显卡 + 本地部署”的经验，在前面的部署试错笔记里被反复强调过。

### `--trust-remote-code`

不少国内的模型需要这个参数，否则 tokenizer 或模型自定义代码无法加载。

# 七、验证 vLLM 是否真的启动成功

不要只看终端“好像没报错”，要做接口验证。

## 1. 看模型列表

```bash
curl http://127.0.0.1:8001/v1/models
```

如果能返回模型信息，说明 HTTP 层已经通了。

## 2. 发一个最小聊天请求

```bash
curl http://127.0.0.1:8001/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{
    "model": "qwen3.5-int4",
    "messages": [
      {"role": "user", "content": "你好，请做个自我介绍"}
    ],
    "max_tokens": 128,
    "temperature": 0.7,
    "stream": false
  }'
```

如果这里能返回答案，你就完成了第一阶段：

> 模型启动成功，vLLM 接口可用。

# 九、项目目录怎么搭


这样以后继续扩展时不会乱。

# 十、第三个阶段：写 `vllm_client.py`

这一步的目标是：**把“调 vLLM”这件事单独封装出来。**

为什么要这样做？

因为后面：

- 换模型名
- 改 base_url
- 调参数
- 加新的模型调用方式

都只用改这里，不用改整个项目。

核心思路是：

- `chat_text()`：负责文本请求
- `chat_image()`：负责图片请求

这一层不负责前端，不负责页面，只负责“和模型对话”。

---

# 十一、第四个阶段：写 `main.py`

这一步开始真正定义你的项目接口。

我们把接口拆成：

- `GET /`
- `GET /health`
- `POST /api/chat/text`
- `POST /api/chat/file`
- `POST /api/chat/image`

## 为什么这 5 个接口很合适

### `GET /`

告诉别人这个服务在干嘛。

### `GET /health`

做健康检查。

### `POST /api/chat/text`

只处理纯文本。

### `POST /api/chat/file`

只处理文件理解。

### `POST /api/chat/image`

只处理图片理解。

这个拆法的意义很大：

- 演示时清楚
- 前后端协作清楚
- 安卓 App 接入时也清楚

---

# 十二、为什么文件接口不直接上传 PDF/Word

因为初学阶段，先做“最容易打通的一条链路”。

最简单的文件理解链路其实是：

1. 先只支持文本类文件
2. 浏览器直接读出文本
3. 发给后端
4. 后端拼 prompt
5. 模型回答

这样做避免了你一开始就陷入：

- PDF 解析
- OCR
- Word 转文本
- 表格抽取
- 编码问题

所以最初只支持：

- txt
- md
- json
- csv

这是一个非常合理的工程取舍。

---

# 十三、为什么图片接口要先压缩再上传

因为手机拍出来的图太大了。本地测试环境有限，如果直接把原图 base64 发给后端：

- 请求体会很大
- 传输慢
- 手机浏览器容易卡
- 服务端也更吃力

所以我们在浏览器里先做：

1. 读取图片
2. 缩放到最长边 768
3. 压成 JPEG
4. 转 base64
5. 再发给后端

这个设计非常适合移动端。

# 十四、第五个阶段：写前端页面 `index.html`

- 调 `/api/chat/image`

# 十五、为什么前端要支持流式和非流式

这是很多人第一次写大模型前端时会遇到的问题。

## 非流式

优点：

- 简单
- 稳定
- 好调试

缺点：

- 要等模型全部生成完才能看到答案

## 流式

优点：

- 更像真实聊天产品
- 用户体验更好

缺点：

- 前后端都要多一层处理
- 解析和渲染更复杂

建议实现方式如下：

- 页面右上角加一个切换框
- 初学时先看非流式
- 跑稳后再体验流式

# 十八、第八个阶段：手机浏览器访问

这一步是整个项目最有“展示感”的部分之一。

## 条件 1：手机和电脑在同一局域网

例如都连同一个 Wi-Fi。

## 条件 2：服务监听 `0.0.0.0`

如果你只监听 `127.0.0.1`，手机访问不到。

## 条件 3：找到电脑局域网 IP

Linux / WSL 常用：

```bash
hostname -I
```

Windows 常用：

```bash
ipconfig
```

假设你的电脑 IP 是：

```text
192.168.xxx.xxx
```

那么手机访问：

```text
http://192.168.xxx.xxx:9000/web/index.html
```

## 如果打不开怎么办

优先排查：

1. FastAPI 是否监听 `0.0.0.0`
2. vLLM 是否监听 `0.0.0.0`
3. Windows 防火墙是否拦截
4. WSL 是否需要端口转发

在你的实习与部署记录里，也明确写到了这条从 WSL 到 Windows 再到手机网页访问的排障路径。

# 十九、你会遇到的典型问题

## 问题 1：模型启动就爆显存

原因通常是：

- 模型太大
- `gpu_memory_utilization` 太高
- `max_model_len` 太大
- 默认显存预留太激进

解决思路：

- 换更小模型
- 降低显存利用率
- 降低上下文长度
- 使用量化模型
- 必要时启用 `--enforce-eager`

用户上传的部署记录里，前面很大一部分内容就是围绕这件事展开：从 1.5B 到 7B、从 FP16 到 AWQ、从 vLLM 到 LMDeploy，对显存、KV Cache 和框架策略差异做了详细试错。

## 问题 2：本机能访问，手机不能访问

大概率是网络层问题，不是模型问题。

优先检查：

- 监听地址
- 防火墙
- 端口转发
- 局域网是否一致

## 问题 3：图片接口报错

要分两种情况：

### 情况 A：前端处理报错

通常是图片读取/压缩失败。

### 情况 B：后端模型报错

通常是底层模型本身不支持视觉输入。

也就是说：

> 接口格式正确，不代表模型一定能看图。

(本项目测试的模型为多模态模型，支持图片分析。)

## 问题 4：文件理解效果一般

因为当前版本是最小实现：

- 只支持文本文件
- 只截前 6000 字符
- 不做分块摘要

这不是 bug，而是有意简化。

# 二十三、学完这个项目，你应该学会...

至少要学会这几件事：

## 1. 模型服务不是网页

网页只是最外层展示。

真正核心是：

- 模型能否启动
- 接口能否稳定提供能力
- 工程结构是否清晰

## 2. 部署不是只会敲启动命令

真正的部署工程还包括：

- 参数选择
- 显存限制
- 接口封装
- 联调排错
- 多端访问验证

## 3. 工程上要先做“小而完整”

不要一开始就追求：

- 超大模型
- 超复杂前端
- 一堆高级功能

先做一个小而完整的项目，才最容易形成正反馈。
